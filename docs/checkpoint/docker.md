1. What is a container and how is it different from a virtual machine?
A container packages an application and all its dependencies together so it runs consistently anywhere Docker is installed. Unlike a VM, a container doesn't include a full operating system: it shares the host OS kernel and only contains the application runtime and libraries it needs. This makes containers much lighter (megabytes vs gigabytes), faster to start (seconds vs minutes), and more efficient at sharing hardware. The trade-off is less isolation: containers share the host kernel, so a kernel vulnerability potentially affects all containers on that host, whereas VMs have fully separate kernels.

2. What is a Dockerfile and what does the build process do?
A Dockerfile is a recipe for building a container image. Each instruction creates a layer: FROM sets the base image, RUN executes commands, COPY adds files from your machine, EXPOSE documents which port the app uses, CMD defines what runs when the container starts. docker build reads the Dockerfile and executes each instruction in sequence, producing an image: a snapshot of a filesystem with your app and all its dependencies baked in. docker run then creates a container from that image, which is a running instance of it. The image is immutable and portable: the same image runs identically on any machine with Docker.

3. Why does your frontend need to know the backend URL at runtime? How did you handle this?
The frontend runs on webserver1 or webserver2 but needs to make API calls to the backend on appserver at 192.168.56.14:8080. I can't hardcode this IP into the frontend image at build time because the same image might be deployed to different environments with different backend IPs. Instead I pass it as an environment variable at container startup:
docker run -e BACKEND_URL=http://192.168.56.14:8080 vitals-frontend:latest

The Node.js server reads this environment variable and exposes it to the browser via a /config endpoint, which the frontend JavaScript fetches on page load. This pattern, externalising environment-specific config, is fundamental to portable containerised applications.

4. What does --restart always do on a docker run command and why would you use it?
--restart always tells Docker to automatically restart the container whenever it stops, regardless of the exit code, including after the Docker daemon itself restarts (i.e. after a server reboot). Without this, if the appserver VM reboots, the backend container stays stopped and the application goes down until someone manually runs docker start. With --restart always, the container comes back up automatically with the VM. In production I'd typically use --restart unless-stopped instead, same behaviour but it respects manual docker stop commands, so you can stop a container intentionally without Docker immediately restarting it.

5. Explain the rollback pattern in your CI/CD deploy steps. Why rename instead of just stopping?
The pattern is: stop running container, rename it to -old (keeping it intact), build and start a new container, health check, then either delete the -old backup if healthy or restore it if not. You should rename rather than delete because docker rename is instant and reversible, the old container with its state is still there, just under a different name. If you deleted it and the new deploy failed, you would have nothing to roll back to, you would have to rebuild from scratch. The rename gives you a safety net that costs nothing while the new container is being health checked. It's a simplified blue-green deployment pattern.

6. Your Go backend reads /proc directly. What happens when you run it in a Docker container?
Containers share the host kernel, so /proc inside the container is the same as the host VM's /proc. The Go backend running in a container on appserver reads appserver's CPU, memory, and uptime, same as if it ran directly on the VM. This is actually a subtle point: the metrics reflect the VM, not the container's resource usage. If you wanted container-specific metrics (how much of its CPU allocation is the container using) you would need cAdvisor or Docker stats. For my use case, monitoring the infrastructure VM's vitals, reading /proc directly is correct and intentional.
