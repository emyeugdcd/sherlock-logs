1. What is a virtual machine and how is it different from a physical server?

A virtual machine is a software-emulated computer that runs inside a host machine. A hypervisor (like VMware Fusion in my project) sits between the hardware and the VMs, allocating CPU, memory, and disk to each one. Multiple VMs share the same physical hardware but are fully isolated from each other: one VM crashing doesn't affect the others. A physical server is dedicated hardware with no sharing. VMs are slower than bare metal due to the virtualisation overhead, but the flexibility, cost savings, and ability to spin up and destroy environments quickly makes them standard for development and increasingly common in production too.

2. What is /proc and why does it matter for system monitoring?
/proc is a virtual filesystem that the Linux kernel maintains in memory. It's not real files on disk: the kernel generates the content on the fly when you read them. It exposes live system state: /proc/stat for CPU tick counters, /proc/meminfo for memory breakdown, /proc/uptime for seconds since boot, /proc/cpuinfo for CPU model and core count. My Go backend reads directly from these files with no external libraries. This is how tools like htop, Node Exporter, and any monitoring agent get their data: they all read /proc. It only exists on Linux, not macOS or Windows.

3. What is Vagrant and what problem does it solve for developers?
Vagrant is a tool for managing reproducible development environments using virtual machines. I define my entire environment in a Vagrantfile: machine names, IPs, RAM, CPU, base OS; and anyone on the team can run vagrant up to get an identical environment. It solves the classic "works on my machine" problem. Instead of manually installing software and configuring settings differently on every developer's laptop, the environment is code. In my project, the Vagrantfile defines 6 VMs with specific IPs and resource allocations, and vagrant up creates all of them automatically.

4. What does apt upgrade dist do and why use it over apt upgrade?
apt upgrade installs newer versions of installed packages but won't add or remove packages to resolve dependencies; it skips upgrades that require dependency changes. apt dist-upgrade (or upgrade: dist in Ansible) is smarter; it will add or remove packages as needed to complete upgrades, handling dependency changes intelligently. For a server I am configuring from scratch and want fully patched, dist-upgrade is the right choice. My Ansible playbook runs this as the first task to ensure the base system is current before installing anything on top of it.

5. What are unattended upgrades and why would you enable them on a server?
Unattended upgrades is a package that automatically installs security updates without human intervention. On a server you configure once and leave running, you can't manually patch it every time a CVE is published. Security vulnerabilities in packages like OpenSSL or the kernel are exploited quickly; often within hours of a public disclosure. Unattended upgrades ensures security patches are applied automatically, reducing the window of exposure.   My Ansible playbook installs it and configures 20auto-upgrades to enable both the package list update and the upgrade itself on a daily schedule.
