1. What is CI/CD and why does it matter?

CI stands for Continuous Integration: every code change is automatically tested and validated as soon as it's pushed. CD stands for Continuous Deployment: code that passes tests is automatically deployed to the target environment. Together they replace the old model where developers would accumulate changes for weeks, merge everything at once, and spend days untangling conflicts and deployment failures. CI/CD makes deployments small, frequent, and automated. Problems are caught immediately when the change is small and fresh in the developer's mind, not weeks later when the code has been mixed with ten other changes. It also removes the "deployment day" as a stressful event: deployments become routine.

2. What is the difference between a job and a step in GitHub Actions?

Steps are sequential commands within a single job: they share the same runner machine and filesystem. If step 3 fails, steps 4 and 5 don't run. Jobs run on separate, isolated machines: each job gets a fresh VM spun up by GitHub. Jobs run in parallel by default, but you use needs: to create dependencies and force sequential execution. The practical implication: you can't share files between jobs without using the upload-artifact and download-artifact actions, because they run on different machines. Within a job, every step can access files created by previous steps because they're on the same machine.

3. What is a matrix strategy and why did you use it for the frontend deploy?

A matrix strategy runs the same job multiple times in parallel with different variable values. In my deploy_frontend job, the matrix defines two servers: webserver1 at 192.168.56.12 and webserver2 at 192.168.56.13. GitHub Actions spawns two parallel job instances, each receiving a different matrix.server value. Both frontend deploys happen simultaneously instead of sequentially, cutting deploy time roughly in half. Without the matrix, I would have either duplicated the entire job block (code smell) or deployed to webserver2 only after webserver1 finished (slower). The matrix also scales cleanly: adding a webserver3 means adding one IP to the matrix array, not duplicating another job block.

4. Why can't your GitHub Actions pipeline actually deploy to your Vagrant VMs? How would you fix it?

GitHub-hosted runners are cloud VMs in GitHub's infrastructure. My Vagrant VMs are on a host-only private network (192.168.56.x) on my laptop. There is no route between a GitHub runner and my laptop's private network: the SSH commands would time out. There are two ways to fix this. First, a self-hosted runner: install the GitHub Actions runner agent on my cicd-server VM, which is inside the same private network as all other VMs. Register it with my GitHub repo and change runs-on to self-hosted. Now the pipeline runs on my own machine with full network access. Second, cloud deployment: move the VMs to a cloud provider like AWS or Azure where they have real IPs and the runner can reach them via the internet. My later projects (Cloud Cartographer, Voyager) will take this second path (I think?).

5. What is the difference between infrastructure provisioning and application deployment? Why should they be separate?

Infrastructure provisioning creates and configures the machines: installing the OS, networking, security hardening, Docker, Nginx. This changes rarely, maybe once a month. Application deployment puts my code onto those machines: building Docker images, running containers, health checking. This happens constantly, potentially dozens of times a day. Mixing them couples two cycles with very different frequencies and risks. A code change shouldn't trigger a full Ansible run that could accidentally modify firewall rules. An infrastructure change shouldn't require a code deploy to take effect. In my project, Vagrant + Ansible handle provisioning (run once) and GitHub Actions handles deployment (runs on every push). In production this maps to Terraform + Ansible for infra and GitHub Actions or ArgoCD for apps.  

6. You have Jenkins installed but unused in your project. Compare Jenkins and GitHub Actions — when would you choose each?

GitHub Actions is a managed service: GitHub runs the infrastructure, I write YAML. It's fast to set up, has a huge marketplace of pre-built actions, integrates natively with my repo, and is what most new projects choose today. Jenkins is self-hosted: I run the server, manage plugins, maintain Java, handle backups. It's operationally heavier but gives me full control: it runs inside my network so it can reach private infrastructure (which is actually relevant to my Vagrant setup), I can run it air-gapped, and I can customise it deeply with plugins. Choose GitHub Actions for new projects, open source work, or when my infrastructure is cloud-based. Choose Jenkins when I'm in a regulated environment requiring on-premises execution, when I have existing Jenkins investment and pipelines, or when I need to reach private infrastructure that can't be exposed to cloud runners.
