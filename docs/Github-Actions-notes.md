# GitHub Actions: Deployment Breakdown & Troubleshooting

## 1. The SSH Timeout "Bug" Explained

If you saw an error like this in your GitHub Actions logs:
`ssh: connect to host 192.168.56.14 port 22: Connection timed out`

**This is not a syntax error; it's a networking reality.**

- **The Runner:** GitHub Actions executes your workflow on a "runner" (a cloud virtual machine hosted in one of GitHub's data centers).
- **The Target:** You are trying to SSH into `192.168.56.14`. This is a *private, host-only IP address* generated by Vagrant on your local laptop. 
- **The Problem:** The public internet (GitHub) cannot route traffic to your private home network. It's like trying to mail a letter to "Bedroom 2" without specifying the country, city, or street address. The connection simply times out.

**How to fix this in the real world?**
1. **Self-Hosted Runners:** You can install a GitHub Actions runner directly on your Mac or inside one of your VMs. Since it's on the same network, it can ping `192.168.56.14`.
2. **Public IPs:** In a real cloud environment (like AWS), your Load Balancer or Web Servers would have Public IP addresses that GitHub can natively reach.

## 2. What do the Deployment Jobs actually do?

Let's break down the `deploy_backend` and `deploy_frontend` jobs to understand the syntax and commands.

### The Syntax Structure
```yaml
      - name: Deploy Backend
        run: |
          ssh -o StrictHostKeyChecking=no devops@${{ env.BACKEND_IP }} '
             ... commands ...
          '
```
- `run: |` : The `|` character tells YAML to execute everything below it as a multi-line shell script.
- `ssh`: We initiate a remote connection.
- `-o StrictHostKeyChecking=no`: This skips the "Are you sure you want to continue connecting (yes/no)?" prompt, which would freeze an automated pipeline.
- `devops@${{ env.BACKEND_IP }}`: We log in as the `devops` user to the IP defined in the environment variables.
- `' ... '`: Everything inside the single quotes is executed *on the remote server*, not on the GitHub runner.

### The Command Breakdown (Inside the SSH session)

#### 1. Graceful Shutdown
```bash
docker stop vitals-backend-app || true
docker rename vitals-backend-app vitals-backend-old || true
```
We stop the current app and rename it to `-old`. The `|| true` ensures that if this is the very first time we run this (and the container doesn't exist yet), the script doesn't crash and fail the pipeline.

#### 2. The Build & Run
```bash
cd /vagrant/backend && docker build -t vitals-backend:latest .
docker run -d --name vitals-backend-app --restart always -p 8080:8080 vitals-backend:latest
```
- **The `/vagrant` Mystery:** You asked why we `cd` into `/vagrant/backend` when you don't see a `vagrant` folder in your project. This is a Vagrant feature! Vagrant automatically takes the folder where your `Vagrantfile` lives on your Mac (which contains the `backend` and `frontend` folders) and mounts it inside the VMs at `/vagrant`. It's a "synced folder". You can actually type `vagrant ssh appserver` and then `ls /vagrant` to see your files!
- We navigate to the backend folder, build the Docker image, and run it in detached mode (`-d`).

#### 3. The Healthcheck & Rollback
```bash
for i in $(seq 1 10); do
    if curl -s http://localhost:8080/metrics > /dev/null; then
        echo "Healthcheck passed!"
        docker rm -f vitals-backend-old || true
        exit 0
    fi
    sleep 3
done
```
This is a resilience loop. It gives the container 10 attempts (waiting 3 seconds between each) to successfully respond to a `curl` request. If it succeeds, it deletes the `-old` backup and exits successfully (`exit 0`).

```bash
echo "HEALTHCHECK FAILED! Executing Rollback..."
docker rm -f vitals-backend-app
docker rename vitals-backend-old vitals-backend-app
docker start vitals-backend-app
exit 1
```
If the loop finishes 10 attempts and never succeeds, we trigger the **Rollback**. We delete the broken new container, rename the old one back to its original name, start it up, and exit with an error (`exit 1`), which turns the GitHub Action red.
