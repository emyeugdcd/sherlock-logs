# BIG PICTURE

1. What is Infrastructure as Code and why is it valuable?
Infrastructure as Code means your infrastructure is defined in files you can version, review, and automate, rather than configured manually through GUIs or ad-hoc commands. Vagrantfile, Ansible playbook, and GitHub Actions workflow are all Infrastructure as Code. The value is reproducibility (anyone can run vagrant up and get identical environments), auditability (git history shows exactly what changed, when, and why), and disaster recovery (if a VM dies we can re-run the playbook, not document tribal knowledge). It also enables code review for infrastructure changes, a firewall rule change goes through a PR just like a code change, not a ticket to a sysadmin.


2. Walk me through what happens from the moment a user types http://192.168.56.11 in their browser to the moment they see the dashboard.

The browser sends an HTTP GET request to 192.168.56.11 on port 80. The loadbalancer VM receives it. UFW allows port 80 on the loadbalancer. Nginx picks up the request and applies the least_conn algorithm to choose either webserver1 (192.168.56.12) or webserver2 (192.168.56.13), then proxies the request to port 3000 on that server. The Node.js frontend container handles the request and serves the HTML, CSS, and JavaScript files. The browser loads the JavaScript, which calls /config to get the backend URL and which web server it's on, then calls the backend's /metrics endpoint on appserver:8080. The Go backend reads /proc/stat, /proc/meminfo, /proc/uptime, formats the data as JSON, and returns it. The JavaScript renders the data as the medical dashboard. The "Web Server" badge shows which webserver handled the request.

3. What is idempotency and why does it matter in DevOps automation?

An operation is idempotent if running it multiple times produces the same result as running it once. In DevOps automation this is critical because automation runs repeatedly: in CI pipelines, on scheduled triggers, after failures. A non-idempotent script that appends to a config file will corrupt it on the second run. Ansible is built around idempotency: every module checks current state before acting. The apt module checks if a package is installed before installing. The copy module checks if file contents match before writing. The ufw module checks if a rule exists before adding it. This means you can safely re-run your Ansible playbook after a partial failure without worrying about duplicate entries or corrupted configs.

4. Your project has a .gitignore that excludes .vagrant/. Why is this important and what happens if you forget it?

The .vagrant/ folder contains VM disk images, snapshots, and VMware-specific binary files: potentially gigabytes of data. It also contains absolute filesystem paths specific to my own machine, so it's useless to anyone else. If I run git add . without a .gitignore excluding it, git tries to hash and stage all of those binary files, which freezes my   terminal for a very long time. If I actually commit and push it, I bloat the repository permanently (git history is forever), potentially expose SSH private key paths, and make the repo almost impossible to clone. The .vagrant/ folder is always machine-specific and always regenerated by vagrant up — it should never be in version control.

5. Explain the defence-in-depth security approach in your project.

Defence in depth means layering multiple security controls so that a failure in any single layer doesn't lead to full compromise. In my infrastructure: the private network means VMs aren't reachable from the internet at all (perimeter). 
- UFW firewall rules ensure each VM only accepts traffic on its required ports even from within the network (network layer). 
- SSH is restricted to key authentication only, root login is disabled, and only specific users are allowed (authentication layer).
- Fail2Ban blocks IPs that attempt brute force (rate limiting layer). 
- Unattended upgrades patches vulnerabilities automatically (patch management layer).
- A strict umask prevents world-readable file creation (file system layer). An attacker who bypasses one layer; say they get onto the network, still faces all the others.

6. Looking at all three projects together — what would you do differently if you were building this for a real production environment?

Several things. Move to cloud VMs with proper networking so the CI/CD pipeline can actually reach the infrastructure;    this removes the central architectural limitation. Use Terraform instead of Vagrant for infrastructure provisioning, since Vagrant is a development tool and Terraform is what production teams use for cloud resources. Fix the WireGuard peer configuration so inter-VM traffic is actually encrypted. Replace the hardcoded password in Ansible with proper secrets management (Ansible Vault or HashiCorp Vault). Add a container registry (Docker Hub, ECR, or ACR) so images are built once and pulled everywhere, rather than built on each VM from source. Implement proper TLS; HTTPS on the load balancer with a real certificate rather than plain HTTP. Add a staging environment so deploys are tested before reaching production. These aren't criticisms of the learning project; they're exactly the gaps the later projects (Cloud Cartographer, Voyager) are designed to close.
